# Deploying the OX Arena backend to Render.com

The repo ships a **Blueprint** (`render.yaml`) that provisions the API, Redis, and
Postgres together.

## One-click (Blueprint)

1. Push this repository to GitHub.
2. Render Dashboard → **New** → **Blueprint** → select the repo.
3. Render reads `render.yaml` and creates:
   - `ox-arena-api` — Docker web service (built from `backend/Dockerfile`)
   - `ox-arena-redis` — managed Redis (`noeviction`)
   - `ox-arena-db` — managed PostgreSQL 16
4. Environment wiring is automatic:
   - `JWT_SECRET` is generated by Render (strong, persistent)
   - `REDIS_URL` and `DATABASE_URL` are injected from the two data services
5. Click **Apply**. First deploy builds the image and starts the service.

## Initialize the database schema

After the first deploy, apply the schema once (from any machine with `psql` and
the external `DATABASE_URL` from the Render dashboard):

```bash
psql "$DATABASE_URL" -f backend/src/db/schema.sql
```

The schema is idempotent (`IF NOT EXISTS`), so re-running is safe.

## Health checks & lifecycle

- Render probes `GET /healthz` (configured via `healthCheckPath`) for zero-downtime
  deploys and auto-restart.
- The service handles `SIGTERM` for graceful shutdown (drains connections, closes
  Redis/PG) on deploys and scale events.
- `GET /readyz` reports dependency health for debugging.

## Point the Android app at Render

In `android/local.properties`:

```
BACKEND_URL=https://ox-arena-api.onrender.com
```

Rebuild the app. Production traffic is HTTPS/WSS (the app's network-security config
only permits cleartext to local dev hosts).

## Manual deploy (without Blueprint)

1. Create a **Web Service** → **Docker**, root `backend/`, Dockerfile `Dockerfile`.
2. Add a Render **Redis** and **Postgres** instance.
3. Set env vars: `NODE_ENV=production`, `JWT_SECRET=<generate>`, `REDIS_URL`,
   `DATABASE_URL`, `PORT=8080`.
4. Set health check path to `/healthz`.

## Environment variables

| Var | Required | Notes |
|-----|----------|-------|
| `PORT` | no (8080) | Render sets this |
| `NODE_ENV` | yes (`production`) | Refuses to boot in prod with the default JWT secret |
| `JWT_SECRET` | yes | Generate; rotate carefully (invalidates sessions) |
| `REDIS_URL` | yes in prod | Without it the in-memory shim is used (single instance only) |
| `DATABASE_URL` | yes in prod | Without it persistence is skipped |
| `CORS_ORIGIN` | no (`*`) | Restrict for a locked-down deployment |
| `RECONNECT_GRACE_MS` | no (15000) | Reconnect window |
| `QUEUE_TIMEOUT_MS` | no (60000) | Matchmaking timeout |
